理解 Lasso (三)：大表格的稀疏查询证明

  • 作者: Yu Guo@Secbit(郭宇): Founder of Secbit, https://github.com/sec-bit

Lasso 这个名字是 Lookup Arguments via Sparse-polynomial-commitments and the Sumcheck-check protocol, including for Oversized-tables 的缩写。这里面有三个关键词，

  • Sparse Polynomial Commitment
  • Sumcheck protocol
  • Oversized table

本文继续讨论如何利用 Sparse Polynomial Commitment 来构造 Indexed Lookup Argument。但为了能处理 Oversized Table（比如 2128 这样的表格），需要充分利用表格的内部结构。

1. 构造简易 Indexed Lookup Argument

前文介绍了 Sparse Polynomial Commitment，现在回到正题 Lookup Argument。下面是一种 Lookup 关系的表示：

Mt=f​

这里 t 为表格向量，长度为 N，f​ 为查找向量，长度为 m，选择矩阵 M 大小为 m×N。

我们引入三个 MLE 多项式 M(X,Y),t(Y),f~​(X) 来分别编码矩阵 M，表格向量 t，与查找向量 f​， 那么它们满足下面的关系：

y​∈{0,1}logN∑​M(X,y​)⋅t(y​)=f~​(X)

Verifier 可以发送一个挑战向量 r，把上面的等式可以归约到：

y​∈{0,1}logN∑​M(r,y​)⋅t(y​)=f~​(r)

现在， Prover 要向 Verifier 证明上面的求和等式成立，我们会立即想到使用 logN 轮的 Sumcheck 协议，把上面的等式归约到一个新的等式：

M(r,ρ​)⋅t(ρ​)=v′

其中 v′ 为 logN 个求和项折叠之后的值，ρ​ 为 Verifier 在 Sumcheck 协议过程中发送的挑战值。这时 Verifier 要验证上面的等式，就需要 Prover 提供三个 MLE 的求值证明。因为 M 矩阵是一个稀疏矩阵，因此 Prover 可以在协议最开头采用 Spark 协议来承诺 M，然后在 Sumcheck 协议的末尾， Prover 可以花费 O(m+m⋅N​) 的计算量来产生 M(r,ρ​) 的 Evaluation 证明。这远好于 Prover 直接采用普通多项式承诺的开销，O(m⋅N)。

协议细节

公共输入：

  1. Ct​=PCS.Commit(t) ，∣t∣=n
  2. Cf​=PCS.Commit(f​)， ∣f​∣=m
  3. CM(spark)​=Spark.Commit(M)， ∣M∣=m×N

第一轮：Verifier 发送挑战向量 r

y∈{0,1}logN∑​M(r,y​)⋅t(y​)=f~​(r)

第二轮：Prover 和 Verifier 执行 Sumcheck 协议，把上式归约到

M(r,ρ​)⋅t(ρ​)=v′

第三轮：Prover 发送 (vM​,vt​,πM(spark)​,πt​)

  1. (vM​,πM(spark)​)=Spark.Eval(CM(spark)​,(r,ρ​),vM​;M)
  2. (vt​,πt​)=PCS.Eval(Ct​,ρ​,vt​;t)

当然，我们可以通过把 M 整个向量排成一排，得到长度为 m×N 长的一维向量 h，然后把这个向量在一个 c 维的空间中 进行拆分:

M(X(D1​),…,X(Dc​))=i=0∑m−1​val(bits(i))⋅eq​(bits(i),X(D1​))⋅…⋅eq~​(bits(i),X(Dc​))

然后利用 Spark 协议来达到 O(m+ccm⋅N​) 的 Proving Time 复杂度。但是 Prover 在产生 πt​ 时则需要 O(N) 的计算量。那么总体上，Spark 虽然可以有效降低 Prover 的工作量，但是如果表格尺寸 N 非常大，那么 Prover 仍然需要花费大量的时间来计算表格。那么还能不能更进一步呢？像 Caulk/Caulk+, cq 那样让 Prover 的性能开销变为关于 N 的亚线性复杂度。

Lasso 协议正是朝着这个方向迈出了一大步，它甚至不需要像 cq 那样要实现对完整的大表格做预处理。尽管它不通用，只能针对几类特殊的表格，但不少常见的运算都可以证明。

Lasso 的核心思想是，我们能否把表格向量 t 像稀疏向量一样按照多个维度去拆解？如果能像 Tensor Structure 那样，一个巨大的表格可以表示为若干个小表格的运算。这样 Prover 和 Verifier 就可以对多个小表格做 Lookup 证明，那么最终得到的效果就是：看起来我们可以实现一个虚拟的大表格的查询证明。

顺着这个思路往下想，一般情况下表格不可能是稀疏的，不过非稀疏的表格在某些情况下是可以分解的。比如我们在前文提到的异或运算的表格，

i01⋮15​A∥B00∥0000∥01⋮11∥11​​A⊕B0001⋮00​​

直觉上，一个2-bit XOR 表格是可以分解为两个 1-bit XOR 表格的运算。因为 XOR 运算是按位进行，操作数的高位和低位的 XOR 运算互不干扰。进而我们可以推广到 AND 运算，OR 运算等等。具体怎么做到呢？接下去，我们深入到表格的内部结构中。

2. 分解表格

稀疏的选择矩阵 M 还有一个特点是，其中的所有非零元素都为 1。那么我们可以换一种方式来表达 Lookup 等式：

f​(X)=i=0∑m−1​T[col(i)]⋅eq​i​(X)

为了排版清晰，这里我们换用大写的 T 表示未被分解的大表格，分解出的子表格用小写字母 t 表示，并且用 T[i] 和 t[i] 符号来表示表格中第 i 个元素 ti​。其中 col(i) 表示第 i 行的 1 所在的列坐标，可以看成是 M 矩阵的一种稠密表示。容易验证对任意的 i∈[0,m)，T[col(i)]=fi​，相当于列出表格中的被 M 非零元素筛选出来的元素。因此这个等式可以看成 f~​(X) 的另一种定义，等价于

f​(X)=y​∈{0,1}logN∑​M(X,y​)⋅t~(y​)

我们把 T[col(i)] 单独排成一个向量 h，然后把向量编码成 MLE 多项式，记为 h~(X)。那么通过一个随机挑战向量 r， Lookup 的关系就归约到下面的等式:

f​(r)=?i∈[0,m)∑​hi​⋅eq​i​(r)

根据 Offline Memory Checking 的思路，我们可以证明 hi​ 都读取自表格 T。这样相当于原地踏步，我们为了证明一个 Lookup关系，我们归约到了另一个 Lookup 关系。不过我们是否可以 hi​ 分解到一个二维（或者多维）的子表格上呢？就像 Spark 协议中的 e 向量一样，我们是把 e 所读取的内存 λ 分解成了 λ(x) 和 λ(y)，然后把 e 分解为 e(x) 和 e(y)。然而并不是所有的表格都能像 λ 一样满足 Tensor Structure 的。事实上，绝大部分的表格不满足这个条件。不过幸运地是，尽管他们不满足 Tensor Structure，但是一大类的有用表格可以按照类似的思路处理。

我们先看一个简单但很实用的表格，RangeCheck 表格。当需要证明 0≤x<2k，我们可以构造一个表格 Trange,k​=(0,1,…,2k−1)，如果 x∈Trange,k​，那么说明 x 在 0 到 2k−1 之间。

这个表格 Trange,k​ 可以被分解成两个 2k/2 的 RangeCheck 表格之间的运算:

Trange,k​[i⋅2k/2+j]=trange,k/2​[i]⋅2k/2+trange,k/2​[j]

比如我们假设 k=4，Trange,4​ 定义如下：

Trange,4​=(0,1,2,3,…,15)

另一个 2-bit RangeCheck 表格 trange,2​ 定义如下：

trange,2​=(0,1,2,3)

那么我们可以用下面的矩阵来展示 Trange,4​ 和子表格 trange,2​ 之间的关系：

0⋅22=01⋅22=42⋅22=83⋅22=12​004812​115913​2261014​3371115​​

矩阵的第一行为 t(x)=trange,2​，第一列也为 t(y)=trange,2​，矩阵中的每个单元可以表示为

Trange,4​[i,j]=22⋅t(x)[i]+t(y)[j]

于是矩阵的所有单元构成了 Trange,4​ 的所有元素。

针对 Rangecheck 表格这个特例，我们可以构造一个高效的 Lookup Argument。

3. RangeCheck 表格的 Lookup Argument

我们用 trange2​(X) 和 Trange4​(X) 表示 2bit 和 4bit-RangeCheck 表格的 MLE 多项式，那么它们满足下面的等式：

Trange4​(X0​,X1​,X2​,X3​)=4⋅trange2​(X0​,X1​)+t~range2​(X2​,X3​)

那么 Lookup 关系可以写成下面的形式：

f​(r0​,r1​,r2​,r3​)​=i∈[0,m)∑​Trange4​(bits(col(i)))⋅eq​i​(r0​,r1​,r2​,r3​)=i∈[0,m)∑​(4⋅trange2​(bits(hi)(col(i)))+trange2​(bits(lo)(col(i))))⋅eq​i​(r0​,r1​,r2​,r3​)​

这里 bits(hi)(col(i)) 和 bits(lo)(col(i)) 分别表示 col(i) 的高 2bits 和低 2bits。

同样，我们需要借助向量 e，其中向量元素 ei​ 表示 T~range4​ 在 bits(col(i)),i∈[0,m) 处的取值，因此上面的等式可以转化为：

f​(r0​,r1​,r2​,r3​)=i∈[0,m)∑​ei​⋅eq​i​(r0​,r1​,r2​,r3​)

由于 T~range4​ 的可分解性，我们可以把 ei​ 的高 2bits 和低 2bits 分别抽取出来，它们构成两个向量 e(x) 与 e(y)，分别对应 trange2​ 中的第 i0​ 项和 第 i1​ 项，满足 i=4⋅i0​+i1​。

接下来构造 e(x) 与 e(y) 两个 MLE 多项式，分别编码 e(x) 与 e(y)，那么上面的等式可以转化为：

f​(r0​,r1​,r2​,r3​)​=b∈{0,1}logm∑​(4⋅e(x)(b)+e(y)(b))⋅eq​(b,(r0​,r1​,r2​,r3​))​

由于这个等式是一个求和式，因此我们可以利用 Sumcheck 协议来把上面的等式归约到：

v′=(4⋅e(x)(ρ​)+e(y)(ρ​))⋅eq~​(ρ​,(r0​,r1​,r2​,r3​))

其中 ρ​ 为 Verifier 在 Sumcheck 过程中产生的长度为 logm 的挑战向量。辅助向量 e(x) 与 e(y) 的正确性可以由 Offline Memory Checking 来证明。

类似的，我们可以把 32-bit RangeCheck 表格分解成四个 8-bit RangeCheck 表格， 或者两个 16-bit RangeCheck 表格。

我们用这个可分解表格，构造一个 Lookup Argument，与之前的方案的差异在于，它利用了表格向量的内部结构，可以处理超大的表格。

4. Lasso 协议框架

Lasso 的核心协议是一个类似 Spark 的稀疏多项式承诺，被称为 Surge。对于任意一个查找记录 fi​，假如 fi​ 在主表格 T 中的索引值为 ai​。因为主表格 T 可被分解，比如 T 可以被分解为 c 个子表格。分解维度的数量 c 对应于主表的索引值 i 按照二进制位的拆分。例如：

bits(i)​=dim(0)(i)∥dim(1)(i)∥⋯∥dim(c−1)(i)​

即每一个主表元素 Ti​ 都可以写成关于 c 个子表格 (t(0),t(1),…,t(c−1)) 中的元素的运算：

Ti​=G( t(0)[dim(0)(i)], t(1)[dim(1)(i)],…, t(c−1)[dim(c−1)(i)] )

我们可以写下对于可分解表格的 Lookup Argument 的等式：

f​(X)=?b∈{0,1}logm∑​G(t(0)[dim(0)(b)],t(1)[dim(1)(b)],…,t(c−1)[dim(c−1)(b)])⋅eq​(b,X)

以 32-bit 的 Rangcheck 表格为例，假如我们需要把它分解为四个子表格，这四个子表格完全一摸一样，都是一个 8-bit 的 Rangecheck 表格。那么我们可以写下下面的等式：

Trange,32​[(i0​,i1​,i2​,i3​)(2)​]=224⋅trange,8​[i0​]+216⋅trange,8​[i1​]+28⋅trange,8​[i2​]+trange,8​[i3​]

这里 G(⋅) 的定义如下

G(y0​,y1​,y2​,y3​)=224⋅y0​+216⋅y1​+28⋅y2​+y3​

协议细节

公共输入：

  1. 子表格的承诺：{cm(t~(i)(X))}i∈[c]​
  2. 查询向量的承诺：cm(f~​(X))

第一轮：Prover 计算并承诺 {dim~(i)(X)}i∈[c]​

第二轮：Verifier 发送随机向量 r∈Flogm

第三轮：Prover 计算向量 e(0),e(1),…,e(c−1)

ej(i)​=t(i)[dim(i)(bits(j))],∀j∈[0,m),∀i∈[0,c)

Prover 计算计数器向量（长度为 logm） c(0),c(1),…,c(c−1)，

Prover 计算终状态中的计数器向量（长度为 logN/c） s(0),s(1),…,s(c−1)

Prover 发送向量的承诺 cm(e(0)),cm(e(1)),…,cm(e(c−1))

第四轮：Prover 和 Verifier 运行 Sumcheck 协议，证明下面的等式：

v=?b∈{0,1}logm∑​G(e(0)(b),e(1)(b),…,e(c−1)(b))⋅eq​(b,r)

Prover 和 Verifier 把等式归约到：

v′=G(e(0)(ρ​),e(1)(ρ​),…,e(c−1)(ρ​))⋅eq​(ρ​,r)

第五轮：Prover 发送 (ve,0​,ve,1​,…,ve,c−1​)，以及 (πe,0​,πe,1​,…,πe,c−1​)

  1. ve,i​=e~(i)(ρ​),∀i∈[0,c)
  2. πe,i​=PCS.Eval(E(i),ρ​,ve,i​;e(i)),∀i∈[0,c)

第六轮：Verifier 验证

PCS.Verify(cm(e(i)),ρ​,ve,i​,πe,i​)=1,∀i∈[0,c)

v′=?G(ve,0​,ve,1​,…,ve,c−1​)⋅eq~​(ρ​,r)

第七轮：Prover 和 Verifier 调用 Offline Memory Checking 证明每个 e(i) 的正确性，即每个向量元素 ej​∈e(i) 都是从表格 t(i) 中读取，读取的位置为 dim(i)(bits(j))：

e(i)=(t(i)[dim(i)(bits(0))],t(i)[dim(i)(bits(1))],…,t(i)[dim(i)(bits(m−1))]),∀i∈[0,c)

5. 二元操作表格的分解

除了 RangeCheck 表格之外，还有 AND, OR, XOR 这类按位计算表格也可以按照同样的思路进行分解。例如下面是一个 1-bit AND 表，记为 AND(1)：

i0123​A∥B0∥00∥11∥01∥1​​A&B0001​​

可以看出，这个表格有 4 行，第 i 行的表格元素为 A&B，而表格的索引值的高位为 A，低位为 B。比如 i=2 这一行，i=(10)(2)​ 二进制位的高位为 1，低位为 0，那么这一行的表格元素为 0，表示 1&0=0。假设我们要分解一个 2-bit AND 表格，AND(2), 那么我们可以用下面的矩阵来表示：

0001​000000010​000000010​000000010​101010111​​

矩阵中的每个单元格表示 (A0​,A1​)&(B0​,B1​)，其中 A0​&B0​=Tand,1​[(A0​,B0​)]，A1​&B1​=Tand,1​[(A1​,B1​)] 满足下面的等式：

AND(2)[(A0​,A1​,B0​,B1​)(2)​]=2⋅AND(1)[(A0​,B0​)]+AND(1)[(A1​,B1​)]

因此，我们可以推而广之，对于任意的 W-bit AND 表格，我们可以把操作数 A 和 B 按位拆分成 c 段，每一段查子表格 AND(W/c) 确定 Ai​&Bi​，然后将 c 个运算结果再按位拼装起来。下面写出这个关系等式：

AND(W)(X0​∥X1​∥⋯∥Xc−1​, Y0​∥Y1​∥⋯∥Yc−1​)=2c−1⋅AND(W/c)(X0​,Y0​)+2c−2⋅AND(W/c)(X1​,Y1​)+…+AND(W/c)(Xc−1​,Yc−1​)

代入到 Lookup 关系等式中，我们可以得到：

f​(X,Y)=?a,b∈{0,1}logm∑​GAND​(AND(W/c)[dim0​(a),dim0​(b)],…,AND(W/c)[dimc−1​(a),dimc−1​(b)])⋅eq​((a,b),(X,Y))

代入上面的 Lasso 协议，我们可以构造出对 AND(W) 表格的 Lookup Arugment 方案。

同样我们可以把其它的二元位操作同样按照这样的思路去分解，如 OR 与 XOR。把主表格拆分成 c 段，假设主表格表示两个长度为 W 的二进制数的位运算，那么第 i 个子表格对应主表索引的第 i⋅(W/c) 位到 (i+1)⋅(W/c) 之间的位运算。(dimi​(a),dimi​(b)) 表示两个操作数 X 和 Y 的二进制位在主表格第 i 个维度上的位置索引。
